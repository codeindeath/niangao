# 年糕 PRD 与产品决策对齐检查 v4.3

## 1. 结论

v4.3 已修复 v4.2 中会导致开发走偏的主要偏差。

当前用户 PRD、管理后台 PRD 和技术架构已经能承接本轮产品决策，可以作为下一步实现设计和开发拆解的基准。

仍需注意：推荐权重、批次规模、质量阈值属于第一阶段默认值，后续应通过真实数据和内容生产校准继续调整，但不再是 PRD 与决策的结构性冲突。

## 2. 已修复的关键偏差

### 2.1 聊聊上下文经验召回

已修复：

- 不再固定“自己的 20 / 收藏 20 / 公共 40”的静态配比。
- 收藏经验改为按当前议题相关领域 / 子领域 / 话题召回，最多取收藏时间最近的 50 条。
- 用户自己的经验中存在明显相关内容时，最终候选至少保留 1 条。
- 公共经验只从 ai_citable 或 high_trust 池补充，不强制保底。
- 最终进入 DeepSeek 的候选仍控制在 3-5 条。

对齐决策：

- 聊聊上下文中的经验应相关、动态配比、受 token 上限控制。
- 用户自己的私密经验只用于该用户自己的聊聊。
- 不强行塞公共经验，避免无关引用。

### 2.2 聊聊沉淀经验

已修复：

- “记下这点”不再只写成用户主动点选。
- PRD 已补充 AI 在关键时刻低打扰提示「这点要不要记下？」。
- 提示只在经验表达清楚、用户情绪不强、不打断倾诉时出现，并做限频。
- 从聊聊生成经验仍默认私密，弹窗确认是否公开。

对齐决策：

- 聊聊场景更私密，沉淀经验默认私密。
- 年糕可以陪伴式地帮助用户沉淀，但不能自动把聊天内容变成经验。

### 2.3 推荐逻辑

已修复：

- 快速划走不再写入长期负偏好，只作为当前候选批次的弱多样性信号。
- 推荐排序补充近期议题 / 处境弱信号。
- 原创高质量池占比写成可随原创供给和正反馈逐步提高，而不是永久固定。

对齐决策：

- 第一阶段不做“不感兴趣 / 减少这类”。
- 推荐纠偏先靠正反馈和行为信号。
- 平台精选早期权重更高，普通用户原创逐渐提高。

### 2.4 搜索逻辑

已修复：

- 搜索结果较少时补充「也许相关」分区。
- 弱相关结果必须来自高质量公开池，并至少命中一个相关维度。
- 空结果仍不做热搜、不做搜索建议，只提供换个说法和聊聊入口。

对齐决策：

- 搜索是处境搜索 + 常规关键词搜索。
- 结果用列表，点开后进入本次搜索结果卡片集。
- 搜索不变成热搜或泛推荐入口。

### 2.5 平台经验冷启动

已修复：

- 管理后台 PRD 补入上线前至少 3000 条平台精选经验目标。
- 补入一级领域比例和子领域优先级。
- 补入前 3 批每批 100 条校准，稳定后可扩大到 300 / 500 条主题批。
- 技术架构补入冷启动进度、领域缺口、子领域缺口、可 AI 引用数量、解读覆盖率。

对齐决策：

- 冷启动阶段先有整体储备打底，不能上线后再慢慢生产。
- 领域使用当前项目定义，不重新创造。
- 内容生产由 Mac Agent + DeepSeek + 年糕后台共同完成。

### 2.6 创作者与来源治理

已修复：

- 补入单个创作者冷启动软上限 50 条，特别高价值可到 100 条。
- 超过 50 条需要报告标记，超过 100 条需要人工批准。
- 补入 source_reliability 三档：high / medium / low。
- 明确来源可靠度影响 AI 引用资格、推荐权重、质量分层和复查优先级，但不直接等同经验质量。

对齐决策：

- 记录真正提出经验的创作者。
- 不把入库数量等同于推荐曝光。
- 来源可靠度和经验质量分分开。

### 2.7 AI 功能治理

已修复：

- `chat_topic_title` 统一改为 `chat_topic_classify`。
- 补入 `moderation`。
- 补入 `translation_normalization`。
- 管理后台和技术架构都补入 function_type、key_alias、queue_name 和降级策略。
- moderation 走 user_realtime，不被 content_low 挤占。
- translation_normalization 走 content_low，成本压力大时可暂停。

对齐决策：

- 年糕 App 内全部 AI 功能由 DeepSeek 实现。
- 不同 AI 功能按 function_type 和 key_alias 拆分，便于统计成本和用量。
- 用户实时链路优先，内容生产低优先级可暂停。

### 2.8 质量评分口径

已修复：

- DeepSeek 原始质量分使用 `ai_quality_score`，1-10。
- 系统归一质量分使用 `quality_score`，0-100。
- quality_tier 由归一分、公开适配判断、来源可靠度和人工覆盖共同决定。
- 前台不展示质量分和分层字段。

对齐决策：

- DeepSeek 可以给质量分和分层标签。
- 前台只理解公开 / 私密，不暴露后台运营分层。

## 3. 当前剩余差距

### 3.1 无结构性冲突

本轮复查后，没有发现会直接违背既定产品决策的结构性冲突。

### 3.2 后续需要运营校准，不需要现在继续问用户

以下内容已经有第一阶段默认方案，但上线后应靠数据和内容生产复盘调整：

- 推荐各池配比和权重。
- 冷启动各领域实际产出数量。
- 原创高质量池占比提高节奏。
- 质量阈值 45 / 60 / 75 / 85 是否需要调整。
- 单批 300 / 500 条在实际内容生产中的处理效率。
- experience_interpretation 的成本与生成覆盖率。

这些不是 PRD 缺口，而是运营和数据校准项。

## 4. 可进入下一步的条件

现在可以基于 v4.3 做下一步实现拆解：

- 移动端页面和交互改造。

- Admin Web 六个一级模块改造。
- AI Gateway function_type / key_alias / 队列治理。
- 平台内容生产和冷启动批次流程。
